# תוכנית עבודה - אוצריא + AI

## 1. תיאור המצב הקיים

**אוצריא** היא אפליקציית Flutter/Dart לקריאת ספרי קודש. החיפוש שלה כיום מבוסס על:

- **Tantivy** (מנוע חיפוש Rust) דרך `flutter_rust_bridge` - חיפוש טקסט מלא מאוד מהיר
- חיפוש הוא **מבוסס מילים בלבד** (BM25 + regex) - אין הבנה סמנטית
- אין הבנה של דקדוק עברי, שורשים, נטיות
- `find_ref` הקיים זה זיהוי לפי תבניות regex של שמות ספרים, לא זיהוי AI

**דיקטה** מציעה משפחת מודלים BERT לעברית:
- `dictabert-tiny` (45M פרמטרים) - הבסיס לעברית מודרנית
- `dictabert-tiny-joint` (45M) - 5 משימות במודל אחד: prefix, morph, lemma, syntax, NER
- `dictabert-base` (200M) ו-`large` (400M) - גרסאות חזקות יותר
- `dictabert-heq` - שאלות ותשובות
- `MsBERT` - **מודל מיוחד לטקסטים רבניים וכתבי יד** (זה זהב טהור לאוצריא)
- `neodictabert-bilingual-embed` - מודל embedding דו-לשוני (עברית/אנגלית)
- `nakdimon` - ניקוד אוטומטי
- `dicta-translate` - תרגום עברית↔אנגלית

## 2. אילוצי החומרה

מחשב של אברך בכולל - הנחה סבירה:

| רכיב | טווח סביר | מינימום שאני מתכנן עליו |
|------|------------|-------------------------|
| מעבד | i3 דור 4-12 | i3 דור 6, 2C/4T, 2.3GHz |
| RAM | 4-16GB | 4GB (אבל לא הכל פנוי) |
| דיסק | HDD או SSD זול | 50GB פנוי |
| GPU | אין | אין |
| אינטרנט | יציב אבל לא תמיד | אופציונלי |

**מסקנות:**
1. רק מודלי `tiny` נטענים מקומית (45M פרמטרים, ~200MB RAM כל אחד עם quantization)
2. מודלי `base/large` ו-`MsBERT` רצים בענן דרך Dicta API (כשיש אינטרנט)
3. **lazy loading** - מודל נטען רק כשמשתמשים בו, ונפרק אחרי 10 דקות חוסר פעילות
4. **INT8 quantization** דרך ONNX - מאיץ פי 2-4 על CPU וחוסך כ-75% מהזיכרון
5. אסור לחסום את ה-UI אף פעם - הכל async/background

### 3.2 למה Python sidecar ולא ONNX ב-Rust ישירות?

שקלתי 3 גישות:

| גישה | יתרונות | חסרונות |
|------|---------|----------|
| **Rust+ort+flutter_rust_bridge** | מהיר, חבילה אחת | מודלי דיקטה דורשים `trust_remote_code` (קוד Python מוטמע), המרה ל-ONNX מורכבת. ה-tokenizer של עברית בעייתי ב-Rust |
| **TFLite** | קל לאריזה ב-Flutter | תמיכה גרועה במודלי BERT עבריים, אין tokenizers |
| **Python sidecar** ✅ | תמיכה מלאה בכל מודלי דיקטה, קל להוסיף כלים, אריזה עם PyInstaller | תהליך נפרד, ~150MB תוספת, זמן עליה ראשוני 2-5s |

**בחרתי בסידקאר** כי המטרה היא "כל הכלים של דיקטה" - הוספת מודל חדש = עוד פונקציה ב-Python במקום שבועות של הנדסת FFI.

### 3.3 Lazy loading + פריקת מודלים

```python
# כל מודל נטען רק בקריאה ראשונה
class ModelRegistry:
    _models = {}
    _last_used = {}
    IDLE_TIMEOUT_SECONDS = 600  # 10 דקות

    def get(self, name: str):
        self._last_used[name] = time.time()
        if name not in self._models:
            self._models[name] = self._load(name)
        return self._models[name]

    def janitor(self):  # רץ ברקע כל דקה
        for name, last in list(self._last_used.items()):
            if time.time() - last > self.IDLE_TIMEOUT_SECONDS:
                del self._models[name]
                gc.collect()
```

תוצאה: בזמן שאת/ה לא משתמש בניתוח דקדוקי, הזיכרון שלו משתחרר אוטומטית.

## 4. הכלים החדשים בפירוט

### 4.1 חיפוש סמנטי (Semantic Search)

**איך זה עובד:**
1. בעת אינדוקס - לכל פסקה מחושב vector של 768 ממדים (`dictabert-tiny`)
2. הוקטורים נשמרים ב-Isar (כבר בשימוש באוצריא)
3. בעת חיפוש - הקלט מומר לוקטור, מחפש cosine similarity מול הוקטורים השמורים
4. תוצאות ה-Tantivy (לקסיקלי) ותוצאות הסמנטי משולבות בציון משולב

**יתרון לאוצריא:** "אדם שעבר עבירה בשגגה" ימצא גם פסקאות שמשתמשות בביטויים "חוטא בשוגג", "טעה ועבר", "שכח ועשה" - גם אם המילים שונות לחלוטין.

**אילוץ i3:** אינדוקס סמנטי של כל הספרייה לוקח זמן (אומדן: 6-12 שעות לכל החומרים על i3). פתרונות:
- אינדוקס רק לפי דרישה (כשמשתמש פותח ספר בפעם הראשונה)
- אינדוקס ברקע בעדיפות נמוכה
- שמירה ב-`~/.otzaria/embeddings.isar` כך שמתבצע פעם אחת בלבד

### 4.2 הרחבת שאילתה לפי שורש (Lemma-aware Search)

**הבעיה הנוכחית:** חיפוש "הלך" באוצריא לא מוצא "הולך", "ילך", "הליכה". הפתרון של regex לא מכסה את כל הצורות.

**הפתרון:**
```dart
// במקום השאילתה הגולמית, אנו שולחים אותה דרך
final analysis = await ai.morphAnalyze('הלך');
// תוצאה: {lemma: "ה.ל.כ", pos: "VERB", root: "הלך"}
final variants = await ai.expandByRoot('ה.ל.כ');
// תוצאה: ["הלך", "הולך", "ילך", "הליכה", "מהלך", "תלך", ...]
// → מורצים כ-OR query על Tantivy
```

**מודל:** `dictabert-tiny-joint`, ~50ms על i3.

### 4.3 מציאת ציטוטים אוטומטית (Citation Finder)

זה אחד הדברים שיכולים להפוך את אוצריא למשהו ייחודי באמת.

**הרעיון:** בכל פסקת ראשונים/אחרונים שמופיעה לעיני המשתמש, השירות מסמן אוטומטית מה ציטוט מהמקרא/חז"ל ומה דברי המחבר עצמו.

**איך:**
1. שימוש ב-`MsBERT` (מודל מיוחד לטקסט רבני) - הוא מאומן לזהות בדיוק את ההבדל הזה
2. המודל רץ ב-Dicta cloud (קונטיניטל. 200M+, כבד מדי ל-i3)
3. תוצאות נשמרות ב-cache מקומי - אותה פסקה לא נשלחת פעמיים
4. בלי אינטרנט: fallback ל-`dictabert-tiny` עם heuristic של ציטוטים מבוססת DB מקומית של תנ"ך/משנה

**UI:**
- ציטוטים בצבע אחר (למשל כחול) עם hover שמראה את המקור המדויק
- כפתור "פתח את המקור בכרטיסייה חדשה"

### 4.4 שאלות ותשובות (QA)

**מודל:** `dictabert-heq` (גרסת base, רק בענן ל-i3)

**זרימה:**
1. משתמש פותח ספר ולוחץ "שאל שאלה על הטקסט"
2. כותב בעברית רגילה: "מתי חרב הבית הראשון?"
3. השרת שולח את השאלה + הקטעים הרלוונטיים (לפי חיפוש סמנטי) למודל
4. המודל מחזיר span בטקסט שעונה על השאלה
5. ה-UI מסמן את התשובה ומציג את ההקשר

### 4.5 קישורים אוטומטיים בין ספרים

**הרעיון:** בזמן קריאה בשו"ע, להראות "כך כותב המ"ב על אותו ענין", "ראה גם את הב"י", "פסיקת הרמ"א מנוגדת ל..." - אוטומטית.

**איך:** index סמנטי של כל הספרים. לכל פסקה הצגת top-k פסקאות דומות בספרים אחרים. סף similarity מינימלי כדי לא להציף בקישורים גרועים.

**אילוץ i3:** היה אפשר לעשות זאת על-pull בלבד (כשהמשתמש לוחץ "מה אומרים על זה במקומות אחרים?") במקום לחשב אוטומטית.

### 4.6 ניתוח דקדוקי, NER, ניקוד, תרגום

כולם פועלים אותו דבר: API call → מודל → תוצאה. ראה `dicta_service/main.py` להגדרות המדויקות.

## 6. מה כבר ראיתי ולא לבזבז זמן עליו

- ❌ **המרת dictabert-joint ל-ONNX מלא** - הוא משתמש בכמה ראשי משימות עם `trust_remote_code`. אפשר להמיר רק את ה-encoder וה-heads בנפרד אבל זה הרבה עבודה. מספיק לקרוא לו דרך transformers ב-Python.
- ❌ **TFLite** - אין הרבה תמיכה בעברית.
- ❌ **לחבר Tantivy ל-Faiss/HNSW** - מורכב. יותר פשוט להחזיק vectors ב-Isar (כבר משולב) ולעשות linear scan עם SIMD - על אינדקס של 1M vectors x 384d (אחרי dim reduction) זה ~150ms על i3, מספיק טוב.
- ❌ **לטעון את `dictabert-large` מקומית על i3** - 1.5GB+ RAM רק למודל, לא פרקטי.
